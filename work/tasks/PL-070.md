## Summary

Added the `locked_at` nullable timestamp column to `teams`, `phases`, `events`, and `odds`, along with `setLocked*`/`clearLocked*`/`getLocked*Ids` data helpers. Taught every sync pipeline (`runStructuralSync`, `runInitialSetup`, `runLiveScoresSync`, `runOddsSync`, and the simulator's `advancePhase`) to skip locked rows. Each sync result now exposes `*Locked` counters for observability.

## Business Spec Coverage

N/A — admin ops infrastructure. Not covered by BUSINESS_SPEC; sits alongside the simulator as an internal tooling primitive.

## Decisions

- **Lock skip lives in the sync layer, not the data layer.** Every pipeline calls `getLocked*Ids` (returns `Set<string>`) alongside its main fetch, then filters in-memory before touching the DB. Data helpers stay generic so `updateEvent` / `upsertPhase` / etc. work the same for both sync and admin edits.
- **Lock is per row, including odds.** Odds lock is keyed on `(eventId, sportsbookId)` (since the odds table is keyed that way). The `getLockedOddsPairs` helper returns `Array<{ eventId, sportsbookId }>`; sync code builds a `"${eventId}:${sportsbookId}"` Set from it.
- **ESPN bridge tables (`external_*`) never lock.** The bridge is the mapping, not the content. Sync continues to refresh `oddsRef`, `statusRef`, etc. on every run so future unlocks pick up fresh URLs. Only the core entity row stays frozen.
- **Locked events are excluded from phase-boundary snap.** A locked event may hold an admin-blessed start time that differs from ESPN's. Rather than read the DB value just to satisfy the snap, the sync skips locked events entirely from the `eventStartTimes` list. The phase snap ignores the ones it doesn't own this sync.
- **Simulator path is treated as a sync pipeline.** `lib/simulator.ts` `advancePhase` calls `getLockedEventIds` and filters before finalizing scores. Without this, running the simulator would silently overwrite any event an admin had locked. Added a test for it.
- **New read helpers shipped but unused in this story.** `getTeamById`, `getTeamsBySportsLeague`, `getPhaseById`, `getEventById`, `getEventsByPhase`, `getOddsById`, `getOddsByEvent`, `updateOdds` are pre-wired for PL-071–PL-074 (read helpers for detail views and admin edits). They're part of the lock infra vertical slice since every later story needs them.
- **New counters added, not replacing existing counts.** `StructuralSyncResult` keeps `phasesUpserted`/`teamsInserted`/`teamsUpdated`/`eventsInserted`/`eventsUpdated`/`eventsSkipped` and adds `phasesLocked`/`teamsLocked`/`eventsLocked`. Downstream shapes (`WeeklySyncResult`, `InitialSetupResult`, `LiveScoresSyncResult`, `OddsSyncResult`) extended the same way.
- **Live-scores sync gets a new "all-locked" skip reason.** If `getScorableEvents` returns rows but they're all locked, the sync short-circuits before the game-window check with `reason: "all-locked"`.

## Deviations from the Story

- None in scope. "Manual edits auto-set locked_at" from the backlog is implicit in PL-072–PL-074 where edit actions land — no action needed in this story.

## Notable / Gotchas

- **Locked phases still exist in `phaseMap`.** In `runStructuralSync`, locked phases short-circuit the `upsertPhase` call but we still have to pick up the existing phase's id so downstream event sync can attach events to it. The code uses `existingPhaseByKey` keyed on `${seasonType}:${weekNumber}`.
- **`upsertExternalPhase` / `upsertExternalEvent` still run for locked rows.** These keep the ESPN→internal mapping current. Only the core `phases.update`/`events.update` are skipped.
- **`getScorableEvents` does NOT filter by `lockedAt`.** First pass embedded the filter in SQL; code review flagged the split policy and it was moved to the sync layer to match the pattern used by `runStructuralSync` and `runOddsSync`. If you add another caller of `getScorableEvents` in the future, remember to apply the lock filter yourself.
- **Odds sync iteration still calls `upsertExternalSportsbook` before the lock check.** This means locked rows still cost one bridge upsert per sync run. Harmless (bridges are idempotent) but worth knowing.

## Follow-ups

- None required for this story. Edit actions (PL-072–PL-074) will consume the new `setLocked*` helpers to auto-lock on admin save.

## Code-reviewer findings (non-blockers not fixed)

- **`T | null` vs `T | undefined` mix across data helpers.** New `get*ById` helpers return `T | null`; other read helpers in the codebase use a mix. Deferred — not in scope for this story, better handled as a dedicated cleanup pass.
- **Locked odds bridge-write cost.** `setup.ts` calls `upsertExternalSportsbook` before checking the lock, so locked rows still incur one bridge write per run. Negligible at current scale; revisit if N grows large.
- **Migration has no backfill or index.** Intentional — locked rows are expected to be rare. Full scan on ~300 events is cheap.
