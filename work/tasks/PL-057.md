## Summary

Replaces the single "Week 1 pick lock" / "phase start" time boundaries with a per-league **start phase** concept (BUSINESS_SPEC §3.8). Each league's start phase is the earliest format-relevant phase whose `pickLockTime` is strictly after the league's activation time (`max(league.createdAt, currentSeason.startDate)`). The start phase's `pickLockTime` is the league's **start lock** — the single boundary that gates joining, inviting, creating a league, and structural edits. A Regular Season league created mid-season simply starts at the next upcoming week instead of Week 1.

## Business Spec Coverage

- §3.1 — League creation gated on start phase existence.
- §3.2 — Structural edits gated on start lock (+ future "no picks yet" gate deferred to PL-028).
- §3.3 — In-season concept narrowed to member removal + leave only.
- §3.8 — New section defining start phase, activation time, and start lock.
- §5.3 / §5.6 — Join / invite gates flip to start lock.
- §13.1 — Updated matrix rows for create + edit-structural.

## Decisions

- **Activation time = `max(createdAt, seasonStartDate)`, not just `createdAt`.** Without the season floor, a league created last June would compute its 2026 start phase based on June 2025 — wrong. Using `max(...)` makes season rollover pick up Week 1 naturally, while mid-season creation still uses `createdAt`.
- **Start lock = pick lock, not phase start.** The "Week 1 open" zone (from phase start to pick lock, ~Tue–Sun in NFL) is when newcomers can still join and be scored against Week 1. Gating at phase start would waste that multi-day window.
- **Remove-member + leave stay on phase start.** Those are membership-state changes that disrupt current members. Unlike structural edits (which only affect future picks), removing someone mid-week mid-submission is worse UX. Narrower gate, stricter freeze.
- **No schema change.** `leagues.createdAt` + `seasons.startDate` are the only inputs. A potential `leagues.startPhaseId` column was considered but rejected — it would need manual re-setting on season rollover and denormalizes a cheap computation.
- **`getPhasesBySeason` replaces `getActivePhasesForSportsLeague` in every call site.** The start-lock check needs ALL phases for the season (to find the earliest eligible one). `isLeagueInSeason` still works with a filtered-to-active subset; the caller now does the filter in app code (`phases.filter(p => p.startDate <= now && p.endDate > now)`). One data query; both checks derived.
- **Picks-absent gate deferred.** §3.2 says "structural edits require start lock unfired AND no picks submitted". Picks table doesn't exist until PL-028 — a `TODO(PL-028)` comment marks the site. Behavior is no-op until picks ship.
- **`leagueActivationTime(createdAt, seasonStartDate)`** is its own exported helper. Simple enough to inline, but keeping it named makes the invariant greppable and the tests obvious.

## Deviations from the Story

- None against the user's intent; broader than the initial "Week 1 open zone" proposal because the full mid-season-start concept is strictly a superset and eliminates the special-case.

## Notable / Gotchas

- **`selectLeagueStartPhase` returns null means two things**: for CREATE → "can't create this league this year"; for JOIN/INVITE/EDIT on an existing league → "you missed the window this season". Both surface as user-facing errors. Once a league's start phase is unset for a season, wait for the next season's rollover to reset activation.
- **`hasLeagueStartLockPassed` returns true when no start phase exists.** Same reasoning — no upcoming lock means no window to act in, so treat as locked.
- **Activation time is computed, not stored.** Every call site must pass `leagueActivationTime(league.createdAt, currentSeason.startDate)`. Skipping this (passing raw `createdAt`) breaks mid-year leagues on season rollover.
- **Tests swap `getPhasesBySeason` mock between future-dated (open) and past-dated (locked) phases.** All action test suites share a default "open" phase fixture (`pickLockTime` in 2099) so base cases pass; locking-required tests override with a past-dated phase.
- **"No picks yet" check is unimplemented.** When PL-028 ships, the structural-edit gate needs `picks.exists({ seasonId, leagueId })` ORed into the lock. `TODO(PL-028)` is in `updateLeagueAction`.
