## Summary

Replaces the single "Week 1 pick lock" / "phase start" time boundaries with a per-league **start phase** concept (BUSINESS_SPEC §3.8). Each league's start phase is the phase matching its configured `(startSeasonType, startWeekNumber)` tuple — and its `pickLockTime` is the league's **start lock**, the single boundary that gates joining, inviting, creating a league, and structural edits.

> **Note (post-PL-058 simplification):** The initial PL-057 design inferred the start phase from `max(league.createdAt, currentSeason.startDate)` ("activation time") because the range was a preset (`seasonFormat = "regular_season"` etc.) and mid-season creators needed the start phase to skip past weeks that had already locked. PL-058 made the commissioner pick the start week explicitly, which made the activation-time shim obsolete — it was removed in the follow-up, along with `leagueActivationTime`. Mid-season creation still works: the action rejects a create when the chosen start week's pick lock is already in the past.

- §5.3 / §5.6 — Join / invite gates flip to start lock.

## Decisions

- **Start lock = pick lock, not phase start.** The "Week 1 open" zone (from phase start to pick lock, ~Tue–Sun in NFL) is when newcomers can still join and be scored against Week 1. Gating at phase start would waste that multi-day window.
- **(Revised post-epic) Remove-member + leave now use the same start-lock gate as every other mutation.** The original PL-057 design kept them on the phase-start gate (stricter) to avoid "disrupting current members mid-submission". That rationale didn't survive review: during the first week's open zone (phase start → pick lock) picks haven't locked yet, so roster changes can't disrupt anything. Collapsing onto one boundary is simpler and more consistent with invites / structural edits. Between-phase gaps are now frozen too, which is a minor tightening but matches how invites already behaved.
- **No schema change.** `(startSeasonType, startWeekNumber)` from the league row + the season's phase list are the only inputs. A `leagues.startPhaseId` column was considered but rejected — it would need manual re-setting on season rollover and denormalizes a cheap lookup.
- **`getPhasesBySeason` replaces `getActivePhasesForSportsLeague` in every call site.** The start-lock check needs all phases for the season so it can find the configured start tuple. `isLeagueInSeason` still works with a filtered-to-active subset; the caller now does the filter in app code (`phases.filter(p => p.startDate <= now && p.endDate > now)`). One data query; both checks derived.
- **Picks-absent gate deferred.** §3.2 says "structural edits require start lock unfired AND no picks submitted". Picks table doesn't exist until PL-028 — a `TODO(PL-028)` comment marks the site. Behavior is no-op until picks ship.
- **(Dropped post-PL-058) `leagueActivationTime(createdAt, seasonStartDate)`** — initial version exported this helper so the activation-time invariant was greppable. Removed entirely after PL-058: with an explicit start week, the phase is resolved by tuple match, no activation arithmetic needed.

## Deviations from the Story

- None against the user's intent; broader than the initial "Week 1 open zone" proposal because the full mid-season-start concept is strictly a superset and eliminates the special-case.

## Notable / Gotchas

- **`selectLeagueStartPhase` returns null when the configured start tuple isn't in the supplied phase list** (e.g. the season's phases haven't been synced yet). Callers treat that as "can't answer yet" — `hasLeagueStartLockPassed` returns `false` in that case so the lock doesn't silently flip to "locked". (Originally it returned `true` when no upcoming in-range pick lock existed; that broke in simulated time because wall-clock `league.createdAt` leaked into the activation computation.)
- **Tests swap `getPhasesBySeason` mock between future-dated (open) and past-dated (locked) phases.** All action test suites share a default "open" phase fixture (`pickLockTime` in 2099) so base cases pass; locking-required tests override with a past-dated phase.
- **"No picks yet" check is unimplemented.** When PL-028 ships, the structural-edit gate needs `picks.exists({ seasonId, leagueId })` ORed into the lock. `TODO(PL-028)` is in `updateLeagueAction`.
