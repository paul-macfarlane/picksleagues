## Summary

Added `updateEventAction` with team picker, UTC start time, status, and score fields, plus an edit dialog on the Events tab of `/admin/overrides`. The schema enforces that home and away teams differ and that scores are provided whenever the status is `final`. Saves auto-set `lockedAt` atomically.

## Business Spec Coverage

N/A — admin ops tooling.

## Decisions

- **Scores stay as strings in the schema; converted via `parseScore` in the action.** Same pattern established by `parseUtcDatetime` for phases — keeps the schema usable on both client (RHF `useForm`) and server (safeParse). Empty string → `null`.
- **Non-negative integer enforcement via regex `/^\d*$/`.** Negative, decimal, and alphabetic inputs fail at the zod layer; the `parseInt` in `parseScore` only runs on values the regex already accepted.
- **Status-to-scores coupling enforced with two refines** (one per field, so the error surfaces on the right input). `superRefine` would collapse them but would also lose RHF's per-field path mapping; kept duplicated by design.
- **Home/away teams must differ**, enforced by schema refine on `awayTeamId`. UI still allows the admin to set both to the same team, but submission fails with a clear field-scoped error.
- **`eventStatusValues` derived from the Drizzle `eventStatusEnum.enumValues`** rather than a hand-duplicated tuple. Code reviewer flagged the duplication; now if a new status is ever added in the schema it flows through to the validator automatically.
- **Team Select uses `Controller`, not `register`.** Radix `Select` is fully controlled (`value` + `onValueChange`) and doesn't work with `register`. Same for the status Select.
- **Teams are sorted alphabetically by abbreviation** in the Select so the admin can find them quickly. Source order comes from `getTeamsBySportsLeague` which has no `orderBy`; sorting is done client-side in the dialog since there are only 32 NFL rows.
- **`getTeamsBySportsLeague` fetched in parallel with `getEventsByPhaseWithTeams`** in `EventsTabContent` via `Promise.all`. The external-event lookup still runs sequentially because it depends on the event IDs.

## Notable / Gotchas

- **`eventStatusValues` is a `ReadonlyArray<string>`** (Drizzle's `enumValues` type). `z.enum` accepts it; if you later need to iterate and pass to a schema that requires a mutable tuple, spread into `[...eventStatusValues] as const`.
- **The Team select allows "no home team" if you click the dropdown without selecting.** In practice RHF's `defaultValues` are populated from the event, so the selects always have a value; but if an event were ever constructed with a missing team, the admin could save one unchanged and the other empty. The zod schema catches this with `z.string().uuid()`.
- **If the admin changes team assignments, the score is not adjusted.** An event with status=final and scores `[home=21, away=14]` will keep those scores even after swapping the teams — admin must update scores to match. UX-wise this is expected for an override tool.

## Code-reviewer findings (non-blockers not fixed)

- **Auth tests per action.** `updateEventAction` (like `updateTeamAction` / `updatePhaseAction`) doesn't explicitly test `UnauthorizedError` / `ForbiddenError` pass-through. The toggle-lock action covers this; the pattern is identical across the update actions. Documenting as known coverage gap.
- **`UpdateEventOutput` comment.** The schema currently has no transforms, so input and output are structurally identical. The three-generic `useForm<Input, unknown, Output>` is used pre-emptively to keep the pattern stable if a transform is added later.
- **External-event fetch sequential after events.** Could be pulled into a follow-up `Promise.all` with a second query; marginal latency impact at 16-event phases.
