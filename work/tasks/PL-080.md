## Summary

Compute and display phase-over-phase standings rank deltas: each user's prior-phase rank vs. current rank, rendered as directional arrows on the leaderboard. Uses on-demand computation over pick results and phases, with natural handling of first phase, mid-phase viewing, and historical seasons.

## Business Spec Coverage

- ¬ß8.3‚Äì8.4 ‚Äî Standings show prior-phase rank delta (arrow + magnitude) under the user's current rank.

## Decisions

- **On-demand computation over snapshot table.** Rationale: simpler (no migration, no invalidation on admin edits), consistent with PL-015 admin-edit flow, bounded query size (~4K rows worst case). User explicitly approved this during planning.
- **Prior phase defined as latest phase ordinal with any scored pick, minus 1.** Naturally handles first phase (empty map), mid-phase viewing, historical seasons, and league range boundaries (PL-057/PL-058) without special cases.
- **Delta sign convention:** `priorRank - currentRank` ‚Üí positive = moved up (favors intuitive reading: "up is good").
- **Removed defensive `continue` on unreachable `priorRank === undefined` branch** per code-review feedback; replaced with `!` + invariant comment.

## Deviations from the Story

None.

## Notable / Gotchas

- The cutoff uses _scored_ ordinals only ‚Äî unscored picks in an in-progress phase don't advance the cutoff (so the delta reads as "rank vs. end of last week" even mid-current-week).
- Every user in `currentStandings` is seeded into `priorResultsByUser` with an empty bucket, so late-joining or zero-score users still land in the prior ranking (tied at 0 points at the last rank) rather than being omitted.
- Also shipped a prettier drift fix on `work/tasks/PL-081.md` (single emphasis marker, unrelated to PL-080).
