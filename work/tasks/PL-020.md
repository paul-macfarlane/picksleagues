## Summary

First story of the Leagues epic. Adds the `leagues`, `league_members`, and `league_standings` tables with `season_format` / `pick_type` / `league_role` enums, `createLeagueAction` that wraps the league + commissioner member + zeroed standing in one transaction, a full `/leagues` + `/leagues/create` + `/leagues/[leagueId]` layout with the 5-tab scaffold from BUSINESS_SPEC §12.4 (placeholders for Standings / My Picks / League Picks), and `assertLeagueMember` / `assertLeagueCommissioner` helpers in `lib/permissions.ts`.

## Business Spec Coverage

- §3.1 — All six configurable settings (name 3–50, size 2–20 default 10, picks per phase 1–16 default 5, season formats, pick types, image URL) validated via `createLeagueSchema`.
- §8.6 — Creator's standing initialized with wins/losses/pushes=0, points=0, rank=1 via schema defaults + `insertLeagueStanding`.
- §12.4 — `(app)/leagues/[leagueId]/layout.tsx` renders the five tabs in spec order (Standings / My Picks / League Picks / Members / Settings). Subpages are `<ComingSoon>` stubs until their owning stories ship.
- §13.1 — "Create a league — Any user": only `getSession()` required, no other gate. "View league data — any league member": layout calls `assertLeagueMember` before rendering any child route.

## Decisions

- **Standings schema owned by PL-020**, not deferred to PL-031. PL-020 explicitly says "initialized with 0-point standing"; the unique constraint on `(leagueId, userId, seasonId)` naturally enforces §3.1's "one season per NFL year" for a given member.
- **`lib/nfl/leagues.ts` vs `lib/leagues.ts`**. The backlog said "lib/nfl/scheduling.ts" for `isLeagueInSeason`; I put the league helpers in a sibling `lib/nfl/leagues.ts` because scheduling.ts is already scheduling-only (lock times / phase boundaries). The helpers themselves are sport-agnostic (`seasonFormatToSeasonTypes`, `selectCurrentSeason`, `isLeagueInSeason`). Keeping them under `lib/nfl/` for now since the `SeasonFormat` enum values line up with NFL phase types today; revisit when a second sport lands.
- **`selectCurrentSeason` fallback chain**: active → nearest upcoming → most-recent. Business error only when no seasons exist at all. Creates an edge where an offseason league created while a new season isn't synced yet will attach to last year's season row. The standing gets replaced at rollover (PL-015), so the fallback is invisible once syncs catch up. Tested.
- **Two user-facing business errors on create** (`NFL not set up`, `No NFL season synced`). Both return `ActionResult` (not throw) because an admin needs to see and act on them; they are not auth failures.
- **Image URL normalization**: empty string → null via `.transform().pipe()`, max 2048 chars defensively. Spec is silent on length; flagged as minor spec-bug candidate in spec-tracer.
- **`Select` wired with `useWatch` + `setValue`**. Not RHF's canonical `Controller` pattern, but matches the project's React Compiler guidance (no `watch()`). Good enough; revisit if more selects land.
- **`Omit<NewLeague, "id" | "createdAt" | "updatedAt">`** on data-layer signatures. Blocks callers from passing DB-managed columns without needing a full input type.
- **No business logic in the form.** Label "Picks per phase" matches the schema column and sport-agnostic convention (initial draft said "per week"; corrected after code review).

## Deviations from the Story

- Backlog mentioned `lib/nfl/scheduling.ts` as the home for `isLeagueInSeason`. Landed in `lib/nfl/leagues.ts` alongside `seasonFormatToSeasonTypes` + `selectCurrentSeason` so the three related helpers live together. PL-021 will import from here.

## Notable / Gotchas

- `selectCurrentSeason` uses inclusive-end (`now <= endDate`). Phases use inclusive-start / **exclusive-end** (`now >= startDate && now < endDate`) because the Tue-2AM phase boundaries back-to-back — the exclusive end prevents a phase from being "active" at the exact moment its successor starts. Tests pin both.
- `withTransaction` in `actions/leagues.ts` intentionally does not catch: unique constraint violations or FK errors bubble up as unexpected errors (Sentry / error boundary). The league name has no unique constraint, so double-click creates are not a concern today.
- Form uses `z.coerce.number()` for numeric inputs because `<Input type="number">` produces a string in the submit payload. `UpdateProfileInput` / `UpdateTeamInput` precedent uses `z.input` for the form and `z.output` for the action.
- The `leagues_sports_league_id_idx` index isn't strictly needed until there's more than one sports league, but Drizzle rules require an index for every FK + WHERE column.

## Code-reviewer findings (non-blockers not fixed)

- **`lib/nfl/leagues.ts` is sport-agnostic** yet lives under `lib/nfl/`. Documented above; revisit when second sport arrives.
- **`SEASON_FORMAT_LABELS` / `PICK_TYPE_LABELS` co-located with the validator**. Both depend on the enum, so co-location is intentional. Not a bug.
- **`getLeaguesForUser` does three round trips** (user's league IDs, leagues, all members for counts). Fine at MVP scale; collapse into a single grouped query if league size explodes.
