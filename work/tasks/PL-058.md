## Summary

Replaces the `season_format` enum preset (Regular Season / Postseason / Full Season) with an explicit (seasonType, weekNumber) **schedule range** stored on `leagues`. Users pick the first and last NFL phase they want their league to cover. Because phase identifiers repeat year-over-year, the range carries over to every future season without re-configuration.

## Business Spec Coverage

- §3.1 — Schedule range replaces "Season format" as the primary configuration; common presets documented as defaults.
- §3.2 — Structural edit gate extended to cover any of the four range columns.
- §3.3 — In-season detection rewritten in terms of "phase in range" instead of "phase matches format".
- §3.5 — Season rollover carries the range unchanged.
- §3.7 — Season-state badge reads the range; label example updated.
- §3.8 — Start phase is the phase matching the configured `(startSeasonType, startWeekNumber)` tuple. (Simplified in a follow-up after PL-058 landed: with explicit start weeks, the activation-time shim from PL-057 became obsolete and was removed along with `leagueActivationTime`.)
- §13.1 — Action matrix rows updated to say "schedule range" instead of "season format".
- §14 — Constants table lists the range shape and end-≥-start + endpoints-exist invariants.

## Decisions

- **Four columns, not a JSON blob.** `startSeasonType`, `startWeekNumber`, `endSeasonType`, `endWeekNumber` are all notnull. Indexable, drizzle-friendly, greppable. An ordered-pair JSON shape would have forced every consumer to parse.
- **Lexicographic ordinal function instead of stored ordinals.** `phaseOrdinal(seasonType, weekNumber) = weekNumber` for regular, `100 + weekNumber` for postseason. Pure, inline, obvious. If a future sport broke the "regular always before postseason" assumption we'd revisit.
- **Endpoint existence checked in the action, not the validator.** The validator only enforces structural shape (ints, valid enum, start ≤ end). "Is `(postseason, 4)` a real phase in the current season?" depends on the live phase list — that check lives in `createLeagueAction` / `updateLeagueAction`, which already fetch the phases for the start-lock gate.
- **Edit form keeps the league's current endpoints selectable even when they're not in the current season's phase list.** Avoids a blank dropdown if an endpoint got pruned between seasons. The action's endpoint-existence check still fires if the user changes the range to something unreal.
- **Default end = last regular-season phase.** Most leagues cover Week 1 → Week 18, so defaulting there is the "least-click path" for the common case. Postseason-only leagues switch both ends.
- **`comparePhasesByOrdinal` exported.** The create page and settings page both need a phase ordering. Rather than duplicate the ordinal math in two pages, I exposed the comparator from `lib/nfl/leagues.ts`.
- **Migration 0008 handwritten.** drizzle-kit's interactive prompt (rename-vs-drop for `season_format`) isn't supported in our non-TTY flow. The SQL file backfills existing rows from their prior `season_format` value: `regular_season` → (regular,1)–(regular,18); `postseason` → (postseason,1)–(postseason,5); `full_season` → (regular,1)–(postseason,5). Then drops the column + enum. Reviewed manually before adding to `_journal.json`.

## Deviations from the Story

- None. The user explicitly asked for pick-your-own bounds, default start = earliest not-yet-started week, same start-lock semantics, shipped in-band with this epic. All four points landed as requested.

## Notable / Gotchas

- **Pro Bowl is filtered at sync, but the ordinal (100 + 4 = 104) still exists.** Users never see "postseason 4" because it isn't present in the phase list — the action's endpoint-existence check rejects any attempt to pick it.
- **Postseason week numbers come from ESPN, not from counting rounds.** Wild Card = 1, Divisional = 2, Conference = 3, **Pro Bowl = 4** (filtered), Super Bowl = **5**. `POSTSEASON_LABELS` in `lib/nfl/leagues.ts` is the single source of truth for the round names. Future NFL playoff structure changes have to update both this map and the ESPN sync.
- **`formatLeagueRange` collapses same-start-and-end to a single label.** A Super-Bowl-only league reads "Super Bowl", not "Super Bowl → Super Bowl".
- **The edit form receives the ordered phase list from the page, not raw.** The page sorts once with `comparePhasesByOrdinal`; the form renders as-is. Keeps `"use client"` components free of business-logic helpers.
- **SeasonStateBadge's label changed shape.** Was "2025 Regular Season · In progress", now "2025 · Week 1 → Week 18 · In progress". The year gets its own separator because the range label can already contain an arrow.

## Follow-ups

- **UI polish:** if the league's range is entirely in the past this season, the badge says "Complete" — clear enough for now, but post-MVP we may want a "Rolls over next season" hint.
- **PL-027 / PL-028 / PL-015** all need to filter phases by the league's range when they ship — `isPhaseInLeagueRange(phase, league)` is the helper to use.
- **Validator NFL-specificity:** `weekNumberSchema` currently accepts any positive int. If someone enters `(regular, 42)` it fails at the action's endpoint-existence check, not the validator. Good enough; tightening means baking sport-specific bounds into a sport-agnostic validator.
