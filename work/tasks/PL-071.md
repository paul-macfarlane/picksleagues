## Summary

Added `/admin/overrides` with URL-driven tabs for teams / phases / events / odds, per-entity read-only detail dialogs, and a lock/unlock toggle per row (unlock gated by an AlertDialog warning). Seasons dropdown scopes phases/events/odds; phases dropdown scopes events/odds. `toggleLockAction` dispatches to the `setLocked*` / `clearLocked*` helpers from PL-070. Admin nav link for Overrides lives alongside Simulator in the user menu.

N/A — admin ops tooling. Not described by BUSINESS_SPEC.

## Decisions

- **Tabs are URL-driven (`?tab=team|phase|event|odds`)**, rendered via server component with `<Link>`-based tab bar. Shareable URLs, clean deep links, no client state. Matches the epic plan approved in Phase 1.
- **Filters navigate, not submit.** `SeasonFilter` and `PhaseFilter` are client `Select`s that push a new URL on change. Changing season resets the phase param — a phase id is meaningless across seasons.
- **Read-only detail uses shadcn `Dialog`**, not a sub-route. One `Details` button per row, keeps the page flat.
- **Lock toggle uses AlertDialog only on unlock.** Locking is immediate (click → action). Unlocking fires a warning dialog ("next sync will overwrite…") because it's the destructive direction. Mirrors the `ResetSimulatorDialog` pattern.
- **`toggleLockAction` dispatches per entity.** Single action with `{ entity, id, locked }` input avoids four near-identical actions. The switch maps to `setLocked{Team,Phase,Event,Odds}` / `clearLocked*` from PL-070.
- **`Profile` used in the test without `as any`.** Built a typed `ADMIN_PROFILE` fixture so the mocked `requireAdminSession` resolves with a real `Profile` shape. Code-reviewer flagged an earlier `as any` cast; fixed.
- **Used `PageProps<"/admin/overrides">`** from `next typegen` rather than hand-rolling the prop type. Ran `pnpm exec next typegen` to pick up the new route.
- **Batched ESPN bridge fetch.** `getExternalEventsByEventIds(eventIds)` is called once per request (using `inArray`) instead of N+1'ing through the event list. First pass had a one-query-per-event loop; code reviewer flagged it.

## Deviations from the Story

- Read-only detail is a `Dialog`, not a separate page — approved in the epic plan.
- Added admin nav link for "Overrides" in the user menu (small scope addition called out in the plan).
- No explicit "search" for phases / events / odds. Teams tab has a client-side search (32 rows, `useMemo`); the others rely on the season + phase filter to narrow. Search across events would be noise on top of the phase filter; revisit if a specific admin flow needs it.

## Notable / Gotchas

- **Every table file is currently marked `"use client"`** even though only `TeamsTable` actually uses local state (for the search input). The reason: each row has a detail `Dialog` + `LockToggle` (both client). shadcn's `Table` already carries `"use client"`, so the directive doesn't cost anything tree-wise, but it reads as louder than it is. Splitting into server-rendered tables + client leaves (`RowActions`) is a clean follow-up.
- **`getExternalEventsByEventIds` returns rows filtered only by `eventId`**, with no data-source scoping. Today only ESPN is a source, so this is fine. If a second data source lands, the lookup would collide — wrap with a `dataSourceId` filter then.
- **Season filter lists descending-by-year seasons via `getSeasonsBySportsLeague`.** Renamed from `getAllSeasons` for consistency with `getTeamsBySportsLeague` / `getPhasesBySeason`.
- **`EventsTable` expects a `Map<eventId, ExternalEvent>`** keyed by the internal event id. Passing a Map keyed by external id will silently miss all rows.

## Follow-ups

- **Split table files into server (table markup) + client (row actions) components.** Non-blocker from code review; small incremental win after the epic settles.
- **Add a `getEventsByPhaseWithTeamsAndExternal` (or `with: { externalEvent }`) to eliminate the two-query fetch in `EventsTabContent`.** Harmless at scale; fold into a cleanup pass.
- **Season-agnostic search across seasons** for phases/events — deferred. Current filter-first UX is sufficient.

## Code-reviewer findings (non-blockers not fixed)

- Tables carry `"use client"` at the module level rather than pushing to leaf action components. Noted above.
- Event layout on mobile vs desktop was flagged as needing eyeball; the final layout is a column of filters + table that collapses to a stacked form on narrow viewports and sits side-by-side with the table below on wider ones — verified by code shape, not by hand.
- Test file imports everything statically now (no top-level `await import`) — reviewer flagged the old pattern; fixed in the rewrite.
