## Summary

Added `updateTeamAction` and `updatePhaseAction` with shared zod schemas, plus edit dialogs on the Teams and Phases tabs of `/admin/overrides`. Saves auto-set `lockedAt` atomically in the same `update*` call. UTC datetime strings are parsed by a shared helper; calendar rollovers and out-of-range times are rejected before hitting the DB.

## Business Spec Coverage

N/A — admin ops tooling, not in BUSINESS_SPEC.

## Decisions

- **Auto-lock happens inside the same `updateTeam` / `updatePhase` call**, not as a second `setLockedTeam` write. One round trip, no risk of a partial state where the row is updated but not locked.
- **UTC datetime is a plain text input, not `<input type="datetime-local">`.** datetime-local reinterprets the string in the browser's local timezone, which is unsafe for admin corrections where DB truth is UTC. Instead the schema enforces `YYYY-MM-DD HH:MM` and the action converts via `parseUtcDatetime`.
- **Schema stays string-only; conversion to `Date` happens in the action.** Keeps the same zod schema usable on both client (RHF `useForm`) and server (safeParse). `z.input` is the form type, `z.output` is the transformed type, and `useForm<Input, unknown, Output>` threads the three generics correctly.
- **Calendar-rollover protection on the UTC datetime schema.** The regex alone passes `2025-02-30 00:00` (which `new Date()` rolls to Mar 2) and `2025-09-09 25:99`. Added `isValidUtcDatetime` that re-reads the parsed Date and compares components. Flagged by code review as a correctness hazard for the "correction tool" use case — fixed.
- **Abbreviation restricted to letters only.** Description in the UI said "2–5 uppercase letters"; schema now enforces `/^[A-Z]+$/` after uppercasing so digits / symbols are rejected.
- **`startDate < endDate` enforced via `.refine` at the phase schema level.** Lexical string compare works because ISO UTC strings sort the same as their parsed Dates.
- **`pickLockTime` is NOT constrained to fall within `[startDate, endDate]`.** Intentional — this is an override tool, and admins may want to test scenarios where lock falls outside the phase window. Code review flagged it for explicit decision; leaving unconstrained.
- **`logoUrl` / `logoDarkUrl` optional-URL handling.** Empty string → `null` via the `optionalUrl` schema piece; validates as a URL otherwise. Matches the DB column shape (nullable text).

## Deviations from the Story

- No `datetime-local` input; see the UTC-string decision above.

## Notable / Gotchas

- **`useForm` requires three generics** (`<Input, unknown, Output>`) whenever the zod schema transforms. Passing only `<Input>` triggers a Resolver-type mismatch between what the hook expects to produce (Input) and what zodResolver actually returns (Output).
- **`z.input` vs `z.infer`.** `UpdateTeamInput` / `UpdatePhaseInput` must be `z.input` so the form's default values (empty-string logo URLs, string datetimes) type-check; the action reads the parsed `z.output` from `safeParse(...).data`.
- **If the action ever receives a pre-parsed object** (e.g., from a non-form caller passing `Date` directly for `startDate`), the string regex will reject it. Today only the edit dialogs call these actions, so this is safe; worth knowing if programmatic calls land later.
- **Abbreviation uppercase.** The schema uppercases the input, so the form displays whatever the admin types but DB storage is normalized. If a UI ever needs the exact typed form, add a separate field.
