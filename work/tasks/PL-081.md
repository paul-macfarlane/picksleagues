## Summary

Persist and display the current period + game clock for in-progress events, wired from ESPN scoreboard → `events` table → live-scores sync → pick card. Also added admin override editing of those fields.

## Business Spec Coverage

- §10.3 — Live games show "LIVE" indicator with current scores, period, and clock; final games show "FINAL" indicator.
- §11.1 — Live scores sync now persists period + clock (previously already parsed off ESPN, but discarded before persistence).

## Decisions

- **Dropped the start time from the `final` status line** (was `Final · Sep 9`, now just `Final`). Spec §10.3 only calls for "FINAL"; the phase selector already provides temporal context on historical views, so the date was clutter.
- **Admin override form accepts period + clock for all statuses.** Live-scores sync nulls them automatically via the ESPN status mapping; the admin surface is "full manual control," so no auto-clear on status change. Admin entries only outlast sync updates while the row stays `locked_at`.
- **`parsePeriod` is a separate helper** from `parseScore` in the validators, even though the bodies are identical — period and score are semantically distinct and may evolve separately (range checks, etc.).
- **`formatPeriod` handles OT as period 5 → "OT", 6+ → "OT{n-4}"** — matches ESPN's NFL period numbering.

## Deviations from the Story

- The story said "ESPN client (`lib/espn/nfl/scores.ts`): parse `period` + `clock` off the scoreboard response." That work was already done in a prior sync/ESPN pass — `FetchedEventScore` already carried both fields. This story's real gap was that the sync pipeline *discarded* them before calling `updateEvent`. No change to the ESPN client was needed.
- Migrations were reset into a consolidated `0000_smiling_the_enforcers.sql` as a separate housekeeping action during this story (not part of the feature work).

## Notable / Gotchas

- `Event` is inferred from the schema via `$inferSelect` — widening the table with new nullable columns cascaded into two test factories (`actions/picks.test.ts`, `lib/sync/nfl/standings.test.ts`) that had to get `period: null, clock: null` added. Expect the same pattern for future `events` additions.
- `updateEvent` exact-match assertions in `lib/sync/nfl/live-scores.test.ts` and `actions/admin-overrides.test.ts` had to be updated — if you add another field to the sync payload, those tests will break the same way.
- The admin dialog calls `parsePeriod` rather than reusing `parseScore`; keep them independent if score parsing ever grows a range check.
