## Summary

Adds the `direct_invites` schema, Zod validators, `data/invites.ts` CRUD + search + listing, three Server Actions (`createDirectInviteAction`, `respondToDirectInviteAction`, `searchInviteCandidatesAction`), a debounced-autocomplete invite dialog on the members page, and an "Open invites" accept/decline list on the home page. Accept inserts the new member + a zeroed standing in a single transaction and auto-deletes the rest of the league's pending invites when capacity is reached.

## Business Spec Coverage

- §5.1 — One pending invite per (league, user); commissioner-initiated; role + expiration (1–30 days, default 7); profile search by username/name.
- §5.3 — Accept validates capacity, not-in-season, not-expired.
- §5.4 — On the join that fills the league, remaining direct invites are wiped.
- §5.6 — Create is blocked when at capacity OR in-season (action + UI).
- §8.6 — Joined member gets a zero standing row keyed to `selectCurrentSeason`.
- §12.2 — Home page renders Open Invites with Accept / Decline buttons.
- §13.1 — Commissioner-only create; invitee-only respond.

## Decisions

- **`upsertDirectInvite` over explicit insert.** The `(leagueId, inviteeUserId)` unique constraint enforces "one pending invite per user per league". `onConflictDoUpdate` lets commissioners refresh an expired row or change the role/expiration without having to delete-then-insert.
- **Inviter FK is `ON DELETE SET NULL`.** The spec wants invite history to survive the inviter's account deletion; the invite still displays to the invitee even if the inviter anonymised their account.
- **"Invite not found" for wrong-owner access.** Ownership check returns a generic business error rather than throwing `ForbiddenError`. This avoids leaking invite existence to third parties — a deliberate security choice, documented for future readers.
- **`searchInviteCandidates` pre-filters excluded users in two parallel queries.** Cheaper than a single giant JOIN at small scale; avoids N+1 and runs both lookups concurrently via `Promise.all`.
- **Autocomplete is the two-allowed-client-fetch exception.** Architecture explicitly calls out "search-as-you-type" as the acceptable client-side fetch pattern; the dialog debounces 200ms and drops stale responses via a `searchSeq` ref.
- **`cleanupInvitesIfFull` runs AFTER the accept transaction.** Spec §5.4 doesn't require atomicity. Keeping it outside the tx avoids doubling the locked-row footprint. If cleanup errors (rare), the invariant is re-established on the next accept — no user-visible inconsistency.
- **Accept revalidates the league layout** because members page (and soon standings) changes.
- **Decline does NOT check in-season/expiration.** A user can always nuke an invite from their home page; reduces friction on cleanup.

## Notable / Gotchas

- **Race on auto-cleanup.** A commissioner who creates an invite in the exact window between "accept commits" and "cleanup runs" will have their invite wiped. Not observable today — both paths go through the same action and our Vercel+Neon setup is low-concurrency — but worth a note.
- **`inviteeUserId` is `text`**, not `uuid`, because BetterAuth user IDs are strings. The Zod validator uses `z.string().min(1)` (not `.uuid()`) for the same reason.
- **`NotFoundError` on `removeDirectInvite`** fires if the invite vanished between `getDirectInviteById` and the delete. Action catches `NotFoundError` and returns a business error ("Invite not found.") when declining, but the happy-path `withTransaction` wraps a second `removeDirectInvite` that could theoretically re-throw `NotFoundError` — the transaction rolls back in that case, which is the right behaviour.
- **ILIKE pattern `%q%` is not escaped**. A user typing `%` or `_` gets wildcards. Low-severity UX polish.
- **`searchInviteCandidatesAction` is commissioner-gated** on the `leagueId`, so a random user can't mine the profile list via the action.
