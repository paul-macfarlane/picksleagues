## Summary

## Decisions

- **Two route groups instead of one `?setup=true` page.** Originally planned a single `/profile?setup=true` dual-mode page, but that required pathname-sniffing middleware to avoid a redirect loop in the layout. Split into `(setup)/setup` and `(app)/profile`; each layout does its own guard (`!setupComplete â†’ /setup` for `(app)`, `setupComplete â†’ /` for `(setup)`). No middleware needed. Reason: Next.js 16 has no first-class API for reading the current pathname in a server layout, and route groups already exist for exactly this kind of guarded shell variant. Context7 confirmed there's no newer API than middleware + custom header for pathname detection.
- **Username uniqueness: pre-check, not error-code sniff.** `actions/profiles.ts` calls `getProfileByUsername` when the username is changing and returns a friendly error if another user owns it. The first pass used `try/catch` on PG error code `23505` + `profile_username_unique`, but the constraint name is fragile (Drizzle auto-naming can change) and matching error messages is brittle. Self-edits skip the check â€” update on the same username is a no-op as far as uniqueness is concerned. The rare race between pre-check and UPDATE will surface as a 500 caught by the error boundary; acceptable for the volume we expect.
- **Avatar sources only from `profile.avatarUrl`.** Previously fell back to `session.user.image` when avatar was null. Better Auth caches `user.image` from the OAuth payload in its user table and never refreshes it, so stale OAuth avatars masked any update to `profile.avatarUrl`. `insertProfile` in `lib/auth.ts` seeds `avatarUrl` from the OAuth image on signup, so new users still see their OAuth avatar until they customize it.
- **`getInitials` is first + last word.** Changed from first-two-words to first + last. "Mary Jane Watson" â†’ "MW" (was "MJ"). Matches the conventional initials format.
- **Layout-level setup guard.** `(app)/layout.tsx` fetches the profile alongside the session and redirects `!setupComplete` users. Every authenticated route inherits the guard automatically â€” no per-page helper to forget. Early draft put the guard in `lib/profile-guard.ts` exporting `requireSetupComplete(profile)` that each page called individually; the self-review flagged this as (a) easy to forget on new pages and (b) a `lib/` layer violation for importing `next/navigation`. Moved into the layout.
- **Avatar preview: `key` on `<Avatar>` root, not on `<AvatarImage>`.** Radix's `Avatar.Root` tracks image-loading status internally; once a successful load flips state to `"loaded"`, unmounting the `<AvatarImage>` doesn't reset it, so clearing the URL leaves the fallback hidden until the page reloads. Keying the Root forces a full remount on every `previewSrc` change and status resets to `"idle"` â†’ fallback renders immediately.
- **`useWatch` instead of `watch()`** in the form for avatar/name preview values â€” eslint's `react-hooks/incompatible-library` flags `watch()` as un-memoizable.

## Notable / Gotchas

- **Typegen after adding routes.** `PageProps<"/profile">` requires `pnpm exec next typegen` to pick up the new route. Any new top-level route needs a typegen pass before typecheck will pass.
- **Action does not revalidate the session.** The auth session's `user.image` remains stale after a profile edit, but the UI no longer reads from it â€” only `profile.avatarUrl`. Don't re-introduce a fallback to `session.user.image` without also refreshing the session (which Better Auth doesn't auto-do for profile edits).
- **`updateProfileByUserId` throws `NotFoundError` if no row matched.** Data-layer narrowing of the destructuring so callers get a non-nullable `Profile`. The action pre-checks `getProfileByUserId` so this only throws on a race.
- **Zod schema allows `""` for `avatarUrl`.** Implemented via `z.union([z.string().url()â€¦, z.literal("")]).optional()` because `z.preprocess` produced an `unknown` input type that broke `zodResolver` typing. The action normalizes `""` â†’ `null` before writing.
- **Username is not lowercased.** PG `text` is case-sensitive, so "Paul" and "paul" currently count as distinct usernames. `generateUsername()` already emits lowercase, so auto-generated usernames never collide by case. Not a spec violation â€” flag for follow-up if it becomes a problem.
- **Per-request dedupe not wired.** `getProfileByUserId` is called in both `(app)/layout.tsx` and every page that needs the profile. Next.js layouts can't pass props to pages; the canonical dedupe is React `cache()`. Left as a cross-cutting follow-up rather than bolted into this story.
