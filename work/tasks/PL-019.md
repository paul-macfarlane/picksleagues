## Summary

Wires league cleanup into the soft-anonymization flow so the BUSINESS_SPEC §2.3 steps that PL-007 deferred to the Leagues epic are finally implemented. Account deletion now blocks sole-commissioners of multi-member leagues up front, deletes sole-member leagues, and removes the user from every other league membership — all inside the same transaction as the identity scrub. Historical standings stay tied to the anonymized `user.id` per §2.3 step 6.

## Business Spec Coverage

- §2.3 step 1 — block when sole commissioner of any multi-member league.
- §2.3 step 2 — delete leagues where the deleting user is the sole member.
- §2.3 step 3 — remove the user from every other league membership.
- §2.3 step 6 — standings survive (keyed to `user.id`, not to `league_members`).

No spec changes needed — the rules were already there.

## Decisions

- **One SQL round-trip for the summary.** `getLeagueMembershipSummaryForUser` returns `{ leagueId, leagueName, role, memberCount, commissionerCount }` per league the user is in, via inline subqueries. Avoids an N+1 `getLeagueMember + getCommissionerCount + getLeagueMemberCount` fan-out over each league.
- **Blocker check is pure.** `getSoleCommissionerBlockers(summary)` lives in `lib/account.ts` as a synchronous filter, reused by both the action (gating) and the account page (displaying the warning).
- **Single transaction for the whole delete.** `anonymizeUser` now accepts the summary and iterates it before scrubbing identity — if any league delete / membership remove fails, the session/account/profile scrub rolls back too.
- **Page-level warning AND action-level check.** Server-rendered account page shows the blocker card and disables the Delete button. The action re-runs the check server-side to prevent skipping the UI. Matches the defense-in-depth pattern used for in-season gates elsewhere.
- **Sole-commissioner message lists every blocking league.** Users with multiple such leagues shouldn't have to delete one-at-a-time to discover the next blocker.
- **`removeLeague` for sole-member, `removeLeagueMember` for everyone else.** Sole-member delete cascades via FK (members + standings + invites); multi-member membership removal leaves the league alone but the user's standing row survives.

## Deviations from the Story

- None; this is a catch-up implementation of §2.3 that was always supposed to land.

## Notable / Gotchas

- **Standings persist on account deletion, get wiped on voluntary leave.** `leaveLeagueAction` (PL-026) clears the user's standings for the league because the user explicitly quit. Account deletion preserves them per §2.3 step 6 — the anonymized tombstone keeps the leaderboard honest. Don't unify these flows.
- **FK cascade on `leagues.id` sweeps `league_standings` for sole-member deletes.** If we ever change that FK to `restrict`, this flow needs explicit `removeLeagueStandingsForUser` inside the transaction.
- **The account page does two round-trips on every load** (session + summary). Fine for the admin-rare "Account" page; don't copy this pattern for hot pages.
- **Blocker list is only consulted at delete time.** A user can sit on the Danger zone card forever; another commissioner may promote themselves out from under them between the page render and the action fire. That's fine — the action re-checks, and the worst case is a stale "blocked" UI that lets the click through with success.
