## Summary

Wired up Better Auth (Google + Discord OAuth via Drizzle adapter) and the DB bootstrap infra it needs. Auto-creates a `profile` row on first login with a randomly generated username. Ships `getSession()` as a ready helper for the `(app)` layout guard that PL-005 will install.

## Business Spec Coverage

- §2.1 Registration — OAuth-only signup, random username generated from email, parsed name from OAuth provider.
- §12.1 Onboarding — auth infrastructure for the login → profile setup flow (PL-005/PL-006 build the UI).

## Decisions

- **Layout-based auth guard, not middleware.** Story listed "Next.js middleware for auth guard on `(app)` routes" but `docs/ARCHITECTURE.md §7` explicitly specifies the layout calls `getSession()` and the error boundary redirects. Architecture doc precedence wins. `getSession()` is shipped here; PL-005 plugs it into `(app)/layout.tsx`.
- **Dual DB driver (`pg` locally, `@neondatabase/serverless` on Vercel).** Picked at module load via `process.env.VERCEL`. `pg` works fine against Neon from Node.js, but neon-serverless is faster in Vercel's serverless runtime due to HTTP/WebSocket pooling. Added `@neondatabase/serverless` in this story (PL-001 had deliberately skipped it — see PL-001.md — but now that we're deploying continuously, we need the driver present).
- **Profile schema + `insertProfile` live here.** The auto-create hook needs both. PL-006 will add validators, setup/edit UI, uniqueness checks, and the rest of the profile data layer. Keeping the minimum here avoids blocking PL-005/PL-006.
- **Dropped PL-002 + PL-003 from the backlog before starting.** See top-of-backlog note: schema + data layer are owned per feature story rather than front-loaded. PL-004 is the first story to consume that pattern.
- **Pulled PL-060 + PL-061 (Vercel + Neon) forward into Foundation.** User wants continuous deploy/test from day one rather than waiting for the end.
- **`generateUsername` kept non-deterministic.** Uses `Math.random()` for the 4-digit suffix; tests assert format via regex. Self-review flagged that passing a `rng`/`suffix` param would make tests fully deterministic — worth doing when PL-006 adds collision-retry logic, not now.

## Deviations from the Story

- No middleware file (see layout guard decision above).
- `(app)/layout.tsx` not created (PL-005 owns route groups).
- Profile validators + setup/edit UI not built (PL-006 owns those).

## Notable / Gotchas

- **BetterAuth `databaseHooks.user.create.after`** is the hook site for profile auto-create. Runs inside the auth transaction — a throw here breaks first login. If the random username collides with an existing one, the unique constraint on `profile.username` throws. Collision probability is low (~0.01% for 10k users on first insert) but PL-006 should add retry-on-conflict before we have real user volume.
- **`.env.example` was already complete** from PL-001 (DATABASE_URL, BETTER_AUTH_URL, BETTER_AUTH_SECRET, GOOGLE_*, DISCORD_*). No env changes needed here.
- **Local docker Postgres has stale `__drizzle_migrations` entries** from prior legacy attempts (different `out` dir). `drizzle-kit migrate` didn't record our 0000 migration, but the tables matching our schema already existed from legacy leftover state. Not blocking — the Vercel/Neon DB from PL-061 starts clean. If we ever need a clean local reset: `docker compose down -v && docker compose up -d && pnpm db:migrate`.
- **`lib/db/schema/relations.ts` not created** — no cross-table reads exist yet. Add it when the first `db.query.*` with relations lands (probably PL-006 when we read profile-with-user).
- **Env vars use `!` non-null assertions** in `lib/auth.ts` and `lib/db/index.ts`. Opaque failure mode if an env var is missing. Good candidate for a boot-time `env.ts` check in PL-060 when we formalize Vercel env management.
