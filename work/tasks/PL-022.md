## Summary

Adds `deleteLeagueAction` (commissioner-gated, business error if the row vanished between read and delete) and a `DeleteLeagueButton` AlertDialog pinned to the settings page's danger zone. On success the client navigates back to `/leagues`. Cascade is entirely FK-driven — `league_members` and `league_standings` already cascade, and future invite/pick tables will carry the same `onDelete: "cascade"`.

## Business Spec Coverage

- §3.6 — Commissioner-only; cascade deletion of associated data.
- §13.1 — "Delete a league — Commissioner" with no in-season condition. The action does not check in-season state.

## Decisions

- **`removeLeague` now throws `NotFoundError`** when the `DELETE … RETURNING id` yields zero rows, matching the existing `updateLeague` pattern. Makes the action's `try/catch` real instead of decorative.
- **Allowed in-season.** Spec says nothing about gating delete on in-season; implemented without a guard. Commissioners are the only ones with access; they bear responsibility for mid-season deletions.
- **Client-side redirect** after success (`router.replace("/leagues")` + `router.refresh`), not `redirect()` from the action. Keeps the action's return contract (`ActionResult`) consistent with the rest of the codebase.
- **Layout-scope revalidation** for `/leagues/[leagueId]` mirrors `updateLeagueAction` so any cached child routes (members/settings/picks) of the deleted league get re-rendered in other tabs.
- **AlertDialog over Dialog.** AlertDialog is the shadcn primitive for destructive confirmations; it forces an explicit choice and traps focus.
- **Dialog copy mentions "invitations"** even though the invite tables don't exist yet. Forward-compatible with the BUSINESS_SPEC §3.6 wording; will be true once PL-023/PL-024 land.

## Deviations from the Story

None.

## Notable / Gotchas

- **Hitting delete on an orphaned league id is unreachable via UI.** `assertLeagueCommissioner` fires first; a user with no membership row gets `ForbiddenError` before `removeLeague` is called. The `NotFoundError` branch only triggers if the row is deleted between the permission check and the delete — tiny race window, but worth the explicit handling.
- **No picks / invites tables yet.** When those stories land (PL-023, PL-024, PL-028+), their schemas MUST set the `leagueId` FK with `onDelete: "cascade"` or the delete will fail with a FK constraint error. The FK cascade is the entire cleanup strategy for this story.
- **`removeLeague` signature change is non-breaking** — all existing callers expect `void`, the throw only fires on a previously-silent no-op path.
