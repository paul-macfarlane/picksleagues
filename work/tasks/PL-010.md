## Summary

Pure ESPN fetcher library under `lib/espn/`. No DB access, no sync logic — PL-011
consumes this slice. Shared client (`espnFetch`, `espnFetchRef`, URL builders,
`EspnApiError`) plus NFL fetchers for seasons, phases, teams, events, live
scores, and odds.

## Business Spec Coverage

Infrastructure for BUSINESS_SPEC.md §11 (ESPN data integration). No user-visible
behavior yet.

## Decisions

- **Ref-based `sports.core.api.espn.com/v2` API, not `site.api.espn.com`
  scoreboard.** Response shapes are stable per-entity records that upsert
  cleanly. Trade-off: more round-trips because every list item is a `$ref` to
  resolve. `docs/BACKGROUND_JOBS.md` was previously documenting the scoreboard
  API — rewrote that section (endpoints, status values, pagination, Pro Bowl /
  week date override copy) to match what we actually use.
- **Status mapping lives in the client.** `fetchEventScore` maps
  `STATUS_SCHEDULED`/`STATUS_POSTPONED` → `not_started`, `STATUS_IN_PROGRESS` →
  `in_progress`, `STATUS_FINAL` → `final`. Tightly coupled to the ESPN response
  shape, so co-locating it here keeps the sync layer clean.
- **Output types use sport-agnostic naming (`FetchedPhase`, `FetchedEvent`)**
  while ESPN-native types keep ESPN naming (`EspnWeek`, `EspnEvent`). Matches
  the schema convention of phases/events over weeks/games.
- **Pro Bowl filter uses `startsWith("Pro Bowl")`.** Catches "Pro Bowl" and
  "Pro Bowl Skills Challenge" both, per BACKGROUND_JOBS.md.
- **No concurrency limiter.** Fetchers loop sequentially over `$ref` items.
  Simple and rate-limit-safe. PL-011 can add `Promise.all` + a pooled map if
  sync runs get slow.

## Deviations from the Story

None. Story listed the exact file layout and all files are present.

## Notable / Gotchas

- **ESPN's `spread` is home-team perspective, not away.** Verified live against
  the 2024 Week 1 KC-vs-BAL opener: ESPN Bet returned `spread: -2.5` with
  `details: "KC -2.5"` and KC (home) flagged `favorite: true`. `fetchOdds` now
  returns `homeSpread = spread, awaySpread = -spread` with the evidence in
  a source comment. The legacy `picksleagues-new-attempt/src/lib/espn/odds.ts`
  had this backwards (`homeSpread: String(spread * -1)`) — do **not** copy that
  code forward.
- **Silent skips in `fetchEvents`.** Events with missing home/away competitors
  are dropped without logging. Fine for the fetch-only slice; PL-011 should add
  structured logging when it wires up observability.
- **`EspnApiError` drops the response body.** Only `statusText` is surfaced
  (often empty under HTTP/2). Including `await response.text()` would improve
  debuggability in sync failures — deferred to PL-011.
- **Tests mock `espnFetch`/`espnFetchRef` at the import boundary.** `client.test.ts`
  is the only file that touches real `fetch` (via `vi.spyOn(globalThis, "fetch")`).
