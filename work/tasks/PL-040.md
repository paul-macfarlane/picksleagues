## Summary

Added `lib/simulator.ts` with `initializeSeason(year)`, `advancePhase()`, `getStatus()`, `resetSeason()`, plus `getAppNow()` (returns simNow when the simulator is active, else real `new Date()`). The simulator drives the existing sync pipelines with a controlled `now`, finalizes events phase-by-phase via a new phase-scoped data helper, and tracks its state in a new singleton table. Also adds `lib/permissions.ts:assertAdmin` (consumed by PL-041) and the `findActivePhase` scheduling helper.

## Business Spec Coverage

N/A — simulator is a dev/testing tool. References `rules/testing.md §The simulator` and BACKGROUND_JOBS §3/§4 (reuses those sync functions).

## Decisions

- **Admin role on `profile.role`** (enum `profile_role` = `user | admin`, default `user`, already in schema from a prior migration). Avoids extending BetterAuth's `user` table. `assertAdmin(profile)` takes a pre-fetched Profile so callers (PL-041's `app/(app)/admin/layout.tsx`) don't need a second query.
- **`simulator_state` is a true singleton** — one row enforced via `singleton INTEGER NOT NULL UNIQUE DEFAULT 1`, upsert targeted on that column. Cleanest PG pattern; no second check constraint or string key needed.
- **simNow plumbed into cron routes via `getAppNow()`** — the four NFL cron route handlers now pass `await getAppNow()` into their sync functions. Sync functions themselves stay pure (`now?: Date`). In practice the cron jobs' existing `isGameWindowActive`/`isNflSeasonMonth` gates mean real cron rarely does work while simulating, but the plumbing is there so "now" is consistent across the app when a simulation is active.
- **Phase-scoped finalization in `advancePhase()`** — new `getScorableEventsForPhase(phaseId, dataSourceId)` in `data/events.ts`; the simulator loops `fetchEventScore` + `updateEvent` directly (parallelized via `Promise.all`). Deliberately **does not** extend `runLiveScoresSync` — that keeps test-only params out of the prod sync entry point.
- **simNow transitions**:
  - Init → `simNow = firstPhase.startDate + 1ms` (first phase is "active").
  - Advance → `simNow = nextPhase.startDate + 1ms` when a next phase exists (skips any between-phase gap, e.g., the conference-championship → Super Bowl lull); `simNow = currentPhase.endDate + 1ms` when advancing past the last phase.
- **Initialization `now = Sept 15 of target year`** — this steers ESPN's `fetchCurrentSeason(now)` to the target season without needing a year-override in `runInitialSetup`.
- **Reset = scoped wipe by deleting the `seasons` row** for the simulated year — PG cascades handle everything else (phases, events, odds, and all `external_*` bridge rows). Teams + external_teams are season-independent and stay.
- **No per-advance odds sync** — odds are synced once at initialization; historical odds don't move.
- **Architecture exception added for `lib/simulator.ts`** in `rules/architecture.md`, alongside the existing `lib/permissions.ts` / `lib/sync/` exceptions. The simulator is structurally identical — orchestrates external fetches + DB persistence + reads its own singleton state.
- **Status display in PL-041 will skip "standings state"** — standings tables don't exist yet (PL-015 / PL-031). Deliberate deviation from the backlog bullet; see Follow-ups below.

## Notable / Gotchas

- **Contiguous phase boundaries + `>` comparison = silently skipped phase.** The first implementation of `advancePhase` computed the next phase via `phases.filter(p => p.startDate > currentPhase.endDate)`. Because adjacent phases share a boundary timestamp (phase N end = phase N+1 start, both snapped to Tuesday 2 AM ET), the strict-greater comparison dropped phase N+1 and jumped to phase N+2. The 2-phase unit test passed by coincidence — the `currentPhase.endDate + 1ms` fallback gives the same arithmetic result as `phase2.startDate + 1ms` when they're equal. Caught by manual testing; fixed in a post-PR commit by walking the ordered phase array with `findIndex + 1` instead of date arithmetic. Regression test uses 3 contiguous phases to force the failure mode.
- **Real cron coexistence**. While the simulator is initialized, real cron jobs will resolve `now` via `getAppNow()` and run against simNow. In practice, `isGameWindowActive`'s 30-minute gate means live-scores and odds-sync rarely fire during typical simNow jumps (advancePhase hops past entire phases). Weekly-sync is idempotent upserts. `setup` is manual-only. If this becomes noisy, short-circuit cron when `getSimulatorState().initialized === true`.
- **Current-year simulation on a prod-shaped DB**. `isValidSimulatorYear` permits `currentYear`. If a simulator row happens to already reference the current year, `initializeSeason` calls `resetSeason` first — which would wipe the real synced current-year data. Practically, no one should be running the simulator against a prod DB; worth flagging in PL-041's UI copy.
- **`getAppNow()` adds one DB read per cron tick.** Negligible in practice, but now a hot-path query.
- **`runInitialSetup` is not transactional** — a failure partway through leaves partial data. The simulator's `resetSeason` is the recovery path. Pre-existing constraint from PL-011.
- **`profile.role` is flipped by hand.** No UI / CLI / action for promotion — deliberate for MVP. Future: a CLI helper or a "promote by email" seed script.
- **`updateEvent` parallelized in `advancePhase`** — divergent from `runLiveScoresSync`, which updates sequentially. Intentional here since the simulator finalizes in one burst; the prod sync's per-event order is not meaningful either, so this pattern could eventually be backported.
