## Summary

Scaffolded the app shell: `next-themes` provider at the root, `(app)` and `(public)` route groups, header with theme toggle and user menu (with sign-out), and a login page wired to Better Auth's Google + Discord social providers.

## Business Spec Coverage

- §2.1 Registration — login page with Google/Discord OAuth buttons.
- §12.1 Onboarding — login is the entry point for first-time users. First-login → profile-setup redirect is PL-006's concern (see Follow-ups).
- §12.2–12.4 — authenticated shell with top header and mobile-first main area.

## Decisions

- **Layout redirect, not error-boundary redirect, for unauth.** `(app)/layout.tsx` wraps `getSession()` in try/catch and calls `redirect("/login")` on `UnauthorizedError`. ARCHITECTURE.md §7 describes the error-boundary variant, but in production Next.js strips error messages, so `error.tsx` can't reliably distinguish `UnauthorizedError` from unrelated bugs. Kept `(app)/error.tsx` for genuine unexpected errors.
- **Sign-out lives in this story.** Not in the original PL-005 bullets but required for a usable authenticated shell — no other story owns it. Added as a user-menu dropdown item (avatar in header → Sign out).
- **User menu takes a narrow `user` prop** (`name`, `email`, `image`) rather than the full BetterAuth session. Keeps the client boundary small and decouples the component from the auth library shape.
- **`getInitials` in `lib/utils.ts`.** Anticipated reuse in member lists, standings, pick attribution. Colocated with `cn()` under the same "general UI helpers" umbrella rather than a one-function file.
- **Inline OAuth brand SVGs in `components/auth/oauth-icons.tsx`.** lucide-react doesn't ship Google/Discord brand marks. Using the official brand SVGs (Google's 4-color G, Discord's blurple logomark) is the standard branded approach.
- **Sports-fan nudge via primary color only.** Set `--primary` to an energetic orange (`oklch(0.62 0.22 34)` light / `0.68 0.2 34` dark). Deliberately minimal — PL-009 owns the real brand system; I only needed a non-corporate accent for this story.
- **`defaultTheme="dark" + enableSystem`.** System preference wins when set; dark is the fallback. Matches the light/dark/system toggle offered by the UI rules.
- **Deleted `app/page.tsx`.** Replaced by `app/(app)/page.tsx` at `/`. The route-group pages don't collide because only one group defines `/`.

## Deviations from the Story

- Sign-out added (not listed in story bullets — see Decisions).
- Login page uses `<Card>` as a container rather than a bespoke hero — simple OAuth-only surface, no need for custom layout.

## Notable / Gotchas

- **`redirect()` after `UnauthorizedError` is reached via a re-throw that is unreachable in practice.** `redirect()` throws `NEXT_REDIRECT` synchronously, so the `throw error` line after the `if (error instanceof UnauthorizedError)` block never runs for non-Unauthorized errors either — it's kept as defensive code for clarity.
- **First-login profile-setup redirect is not wired here.** `authClient.signIn.social({ callbackURL: "/" })` routes everyone to `/`. PL-006 needs to add a `setupComplete` guard in `(app)/layout.tsx` (or `(app)/page.tsx`) that sends first-time users to `/profile?setup=true`. Until PL-006 ships, new users land on the authed home placeholder.
- **`suppressHydrationWarning` on `<html>` is required** for `next-themes` to avoid hydration mismatch warnings on the initial theme class.
- **Typegen cache.** After deleting `app/page.tsx`, stale `.next/**/validator.ts` entries pointed to the removed page. Fix: `rm -rf .next && pnpm exec next typegen`. Worth remembering when renaming/removing top-level routes.
