## Summary

Adds the `/account` page with a Danger Zone that soft-anonymizes the signed-in
user: deletes sessions + linked OAuth accounts, scrubs `user`/`profile` identity
columns, signs them out, and redirects to `/login`. Also relocates the theme
picker from the header into the user menu (radio group with Sun/Moon/Monitor
icons) and keeps a slim standalone toggle for the `(public)` login layout.

## Business Spec Coverage

- §2.3 Account Deletion — soft anonymization flow with session/account wipe and
  profile scrub. Spec §2.3 updated in this commit to document the userId-suffixed
  placeholder values (the literal `"anonymous"` values collided with the
  `user.email` / `profile.username` unique constraints).
- §12.7 Account Management — Danger Zone card with confirmation dialog on
  `/account`.

## Decisions

- **Anonymize by userId suffix, not literal `"anonymous"`.** `user.email` and
  `profile.username` are both `UNIQUE NOT NULL`. Using a literal placeholder
  would work for one deletion and then collide. Chose deterministic suffixed
  values: `anonymous+<userId>@deleted.picksleagues.local`,
  `anonymous-<userId>` (truncated to the 50-char profile username cap). Updated
  BUSINESS_SPEC §2.3 to match — don't update the code back to the literal.
- **Keep the `user` row, don't delete it.** Picks/standings (future) are
  joined to `user.id`; cascading a delete would destroy history. Better Auth's
  `api.deleteUser` would drop the row and cascade the profile, so we scrub in
  place from the data layer instead.
- **League checks deferred.** No league schema exists yet (PL-020+), so the
  sole-commissioner block, sole-member league delete, and membership removal
  can't be implemented now. Those extensions land with their owning stories
  (see _Follow-ups_). Chose this over a permissions-stub file, which would
  violate WAYS_OF_WORKING §7 ("no placeholder or stub implementations").
- **Data functions stay generic.** `updateUserById`, `updateProfileByUserId`,
  `deleteSessionsByUserId`, `deleteAccountsByUserId` — no `anonymizeUser` in
  `data/`. The orchestration lives in `lib/account.ts` per the architecture
  rule that `data/` is generic and business intent stays in `lib/`.
- **Transaction wraps the whole scrub.** `withTransaction` in the server
  action means a failure after sessions are deleted (e.g., the profile update
  hits the unique index) rolls back and the user stays intact.
- **Sign-out failures are swallowed on the client.** Once the server action
  returns success, sessions are gone server-side; the client's Better Auth
  cookie will fail on the next request anyway, and `router.replace("/login")`
  gets the user to the right place. Comment documents the reason.
- **Theme picker moved into user menu.** Header real estate is valuable on
  mobile, and theme is a preference, not a nav action. Retained a standalone
  `<ThemeToggle>` on `(public)/layout.tsx` because unauthenticated pages have
  no user menu. Both use the same radio-group-with-icons pattern for visual
  consistency.

## Deviations from the Story

- Story bullet says "Confirmation dialog" without specifying library; implemented
  with `<AlertDialog>` from shadcn (already installed). Matches the ui rules.
- Story doesn't mention the theme-picker move — folded in because it's a small
  UX improvement in the same header/menu surface PL-007 touches.

## Notable / Gotchas

- **No Zod step in `deleteAccountAction`.** The action takes no input, so the
  canonical validate→auth→authorize→execute→revalidate→return sequence skips
  step 1. This is correct; don't add a no-op schema.
- **`data/auth.ts` is the auth _data_ layer, not a Server Actions file.** Lives
  under `data/`, not `actions/`. Mirror this naming if you add more auth data
  helpers.
- **`next typegen` was required** after adding `app/(app)/account/page.tsx`
  before `pnpm typecheck` would recognise the new route. This is the same
  footgun PL-006 flagged — any new route requires typegen.
- **`THEME_OPTIONS` is duplicated** in `components/layout/user-menu.tsx` and
  `components/layout/theme-toggle.tsx` (six lines). Left inline rather than
  creating a one-off shared module; hoist if a third theme surface appears.
- **Re-login after deletion creates a fresh user.** Signing in with the same
  Google/Discord account after deleting yields a new `user.id` because Better
  Auth's `account` rows are gone. The old `user` row remains as an anonymized
  tombstone. Spec doesn't forbid this, and it's the right default for a
  "deleted" flow.
